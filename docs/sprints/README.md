# Guía de gestión de Sprints, Features y Issues

## Estructura
- **/docs/sprints/**: contiene guidelines, matriz de sprints, checklist de entregables y documentación contractual de cada fase.
- **/docs/sprints/features/**: una subcarpeta por cada feature relevante o vinculante, traceadas por ID.
- **/docs/sprints/issues/**: bugs, mejoras o incidentes contractuales documentados (issues fundamentales deben tener seguimiento aquí, incluso si hay tickets en Github).
- **/docs/sprints/templates/**: plantillas (features, issues, sprint-readme) alineadas a ISO/QA/PO.

## Workflow recomendado
1. Cada cierre de sprint debe tener README checklist, tag y matriz contractual mínima.
2. Features y bugs deben crearse siempre mediante plantilla y vincularse a los requisitos contractuales.
3. Cada feature, issue y entregable debe referenciarse en el CHANGELOG y documentarse su cierre en el handoff si aplica.
